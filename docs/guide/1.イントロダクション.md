### 1.1 この講座で解説すること

この講座では、**2026年時点で、AIプロダクト開発の現場で実際に使われている React SPA のライブラリ構成**を紹介します。

本講座では、実際のAIプロダクト開発現場での経験をベースに、
- 今、現場で選ばれているライブラリは何か
- なぜそのライブラリが選ばれるのか

を、現場のエンジニア視点で解説していきます。

なお適切なアーキテクチャやライブラリは、プロジェクトの規模やプロダクト特性によって異なります。今回紹介する内容は、**あくまで「AI プロダクトの現場では、こうした構成がある」という一例**です。  
内容を参考にしながら、ご自身のプロダクト開発や学習に役立ててもらえればと思います。


### 1.2 AI時代のReact開発

近年の開発現場では、フロントエンドに対して次のような要件が強く求められます。

- **開発スピード**
- **仕様変更への柔軟な対応力**

特にAIプロダクトでは、AIモデルの進化やクライアント要件変更などによって、UIに求められる機能や見せ方が大きく変化することが珍しくありません。「作ってみないと正解がわからない」という不確実性が高いため、**変更に強い構成**が不可欠です。

#### 従来の React 開発の課題

こうした状況下で、従来よく見られた `useEffect` を多用する構成は、開発のボトルネックになりがちです。
例えば、以下のようなコードです。

```tsx
// 従来のデータ取得パターン
export default function ItemsPage() {
  // 1. 状態管理が面倒（データ、ローディング、エラー...）
  const [items, setItems] = useState<Item[]>([]);
  const [isLoading, setIsLoading] = useState(true);

  useEffect(() => {
    // 2. データ取得処理がコンポーネントに混ざる
    const fetchItems = async () => {
      const data = await getItems();
      setItems(data);
      setIsLoading(false);
    };
    fetchItems();
  }, []);

  if (isLoading) return <div>Loading...</div>;

  return (/* ... */);
}
```

このような実装には、主に2つの問題点があります。

1.  **処理の流れが追いづらい**
    *   コンポーネントの中に「表示ロジック」と「データ取得ロジック」が混在するため、コードが複雑になります。
    *   「どの処理が、どの順番で実行されるのか」が直感的に分かりづらくなります。

2.  **実装のバラツキ**
    *   「どこでデータを取るか」「どうやってエラーを処理するか」のルールが曖昧になりがちです。
    *   その結果、画面や担当者によってコードの書き方がバラバラになり、保守性が低下します。

その結果、現状の仕様を理解することが難しくなり、機能追加のたびにバグが生まれるなど、変更の難易度が高くなってしまいます。

本講座では、こうした課題を解決するための **「変更に強い、かつ、開発スピードを高められる」React 構成** を紹介します。


### 1.3 この講座で扱うライブラリと構成の考え方

本講座では、以下のライブラリなどを組み合わせた構成を解説します。

| ライブラリ | 役割 |
|---|---|
| **React Router (Framework Mode)** | ルーティング・データ取得の基盤 |
| **TanStack Query** | 非同期データの管理・キャッシュ |
| **Axios** | HTTPクライアント |
| **Zustand** | グローバルステート管理 |
| **React Hook Form & Zod** | フォーム管理・バリデーション |
| **MUI (Material UI)** | UIコンポーネント |

それぞれのライブラリについて、単なる使い方の解説ではなく、
- **なぜこのライブラリを採用するのか**
- **実際にどのように利用するのか**

を、サンプルアプリの実装を通して具体的に説明します。


### 1.4 対象者・前提知識
